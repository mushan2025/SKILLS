# Biomni Trace Local Repro

Use this skill when the user wants to convert a Biomni share trace into local, reproducible, teaching-oriented bioinformatics scripts.

This skill complements `general-bioinformatics-teaching-script-enhanced`: this skill governs trace download, corrected-path selection, continuity across project scripts, and decision gates; the teaching-script skill governs how to explain code for zero-bioinformatics and zero-coding readers.

## Non-Negotiable Rules

1. **Download first, analyze second.** If the user provides one or more Biomni share links, download the execution trace, code, manifests, notes, tables, figures, and other available files before judging the analysis.
2. **Handle multiple Biomni agent windows.** A single project may have several Biomni share links because the user opened multiple Biomni agent windows for different steps, audits, retries, or later corrections. Treat multiple links as one project evidence set unless the user says they are unrelated. Download each link into a separate trace folder, record source URLs and tokens, and build a cross-trace chronology before deciding the final corrected path.
3. **Use the latest relevant local trace download.** If multiple downloads exist for the same project, prefer the newest relevant evidence set unless the user explicitly asks for an older trace. Do not ignore older links automatically: they may contain context, inputs, or correction history needed to understand later outputs.
4. **Use first-hand local evidence; do not guess.** Understand and write scripts from the downloaded local conversation/context, execution trace, code, manifests, notes, tables, figures, logs, and output files. Do not infer the workflow from memory, filenames, or plausible bioinformatics habits when first-hand trace evidence is available.
5. **Trace paths are evidence only.** Final local reproduction scripts must not use `share_*`, `.tmp_*`, `/mnt/...`, Biomni workspace paths, `biomni_traces/`, or downloaded trace folders as default runtime inputs.
6. **Corrected path only.** Executable code must reproduce Biomni's final corrected path, not every path Biomni attempted. Historical mistakes, deprecated outputs, old formulas, wrong audit routes, and wrong interpretations must appear only in `#` comments.
7. **Preserve project continuity.** Before writing a new script in an ongoing project, read the earlier local scripts and their outputs so the new script connects cleanly.
8. **Local-uploaded originals only.** Raw/original research input files must come from local user-provided files, not from Biomni runtime paths or automatically downloaded public databases, unless the user explicitly approves a different source.
9. **Local-generated intermediates only.** Any intermediate file required as script input must be generated by earlier local reproduction scripts. Do not use Biomni output tables as default intermediate inputs.
10. **Do not run full workflows by default.** Codex writes code, writes Chinese comments, and performs static/parse checks. Do not execute the complete analysis pipeline unless the user explicitly asks for real execution.
11. **Prepare a beginner environment guide before scripts.** Before starting or extending user-facing reproduction scripts, make sure the project has a Word `.docx` guide for zero-background learners that explains which language/runtime the project uses, which software to download, how to install/configure it, how to open the project, how to run scripts, and how to save/reopen the working state. Do not assume every project is R; choose R, Python, notebooks, command-line tools, or other runtimes according to the trace and local project. If the guide is missing or outdated, create or update it before the analysis script work.
12. **Write for absolute beginners.** Assume the learner has zero bioinformatics background and zero coding background. Explain the research question, biological meaning, data objects, language/runtime syntax, file interfaces, algorithms, parameters, and how to judge results.
13. **Chinese teaching comments with readable layout.** User-facing reproduction scripts should use Chinese comments and Chinese explanatory text by default. Do not leave meaningful executable code unexplained, but avoid a wall of text: use section overviews, algorithm cards, and concise inline comments for key syntax or likely misunderstandings.
14. **Version, seed, and code fidelity.** Software versions, language package versions, random seeds, parameters, and code structure should match Biomni as closely as practical. Change only what is needed for local paths, local prerequisites, teaching clarity, and removal of executable error paths.
15. **Ask for major decisions.** If a choice changes the analysis path, candidate definition, default runtime behavior, output interface, naming convention, script deprecation, deletion/overwrite behavior, whether to run expensive analysis, whether to use non-local source files, or whether to deviate from Biomni versions/code structure, ask the user first.

## Workflow

### 1. Download the Biomni share

- Accept one or more Biomni share URLs. Multiple links are common when one project was advanced through several Biomni agent windows.
- Extract the share token or stable identifier from each user-provided URL.
- Find the real downloadable content behind the share page before analyzing. Read `references/link-discovery.md` for the required discovery procedure.
- Prefer running `scripts/download_biomni_share.py` first when Python and network access are available; it saves viewer/API responses, discovered URLs, reachable files, and a manifest.
- For a share URL like `https://biomni.phylo.bio/replay/share_xxx`, the viewer page is only the human-facing entry. Extract `share_xxx`, then use the public replay API routes discovered from the frontend configuration. In recent Biomni shares the API base resolved to `https://api.phylo.bio`, and the important route shapes were:
  - `GET <API_BASE>/v2/public/replays/<share_token>` for the full shared replay/session/context.
  - `GET <API_BASE>/v2/public/replays/<share_token>/results` for result-file metadata.
  - `GET <API_BASE>/v2/public/replays/<share_token>/files/<result_id>/download` for a signed download URL for one result file.
  - `GET <API_BASE>/v2/public/replays/<share_token>/files/<result_id>/download?inline=true` for previewable inline content when supported.
- Treat those API routes as the primary known download entry points, but still save the viewer HTML and JavaScript chunks because the API base or route details can change. If the current frontend exposes a different API base, use the current frontend value and record it in the manifest.
- Save the download under a project-local trace folder such as:
  - `biomni_traces/<share_token>_<YYYYMMDD>/`
  - or the nearest existing project convention if one is already present.
- For multiple share links, use one folder per token and create or update a small cross-trace inventory that records each source URL, token, download date, apparent project step, and how it relates to the other links.
- Download all reachable execution trace materials: trace JSON/markdown, code snippets, result metadata, file manifests, tables, figures, notes, and logs.
- Record the source URL, download date, token, and file inventory.
- If network access is blocked, request approval through the active Codex permission flow.

### 2. Audit the trace

Read enough trace material to establish:

- the final corrected analysis path;
- which files are canonical outputs;
- which files are deprecated or historical;
- which formulas, thresholds, candidate definitions, and interpretations were corrected;
- which local files must be generated by earlier scripts;
- the software versions, package versions, random seeds, parameters, and command/code style used by Biomni;
- which outputs are only audits, intermediate evidence, preliminary shortlists, or user-review candidates rather than final biological conclusions.

Never infer the final path from filenames alone. Prefer trace notes, manifests, result metadata, and corrected output tables.

When multiple Biomni share links belong to the same project, audit them together. Build a chronology across links using conversation timestamps, file creation times, step names, version suffixes, audit notes, and user instructions. Later windows may depend on earlier windows for context; earlier windows may also contain mistakes that were corrected later. Use all first-hand evidence, but implement only the final corrected route.

When the trace contains multiple versions (for example `v21`, `v22`, `v23`, `v24`), do not assume the latest file is automatically the final biological result. Determine what the latest version actually means from first-hand notes and audit files. It may be a file-level validation patch, a preliminary shortlist, or a user-review candidate input for a later step, not the endpoint of the analysis.

### 3. Inspect existing local scripts

Before creating or extending a script, inspect the same project's existing local reproduction scripts.

Build a mental interface map:

- script order, for example `F0 -> F1 -> F2A -> F2B -> F2C -> F2step4`;
- default `PROJECT_DIR`;
- inputs and outputs of each script;
- object/table names used as handoff points;
- stop conditions and user-facing error style;
- terminology and biological narrative already introduced.

The new script must not create a logical gap. It should neither repeat previous modules as if they did not exist nor assume knowledge that earlier scripts never introduced.

Check continuity explicitly:

- every required input in the new script must be produced by local-uploaded originals or earlier local scripts;
- every missing prerequisite should `stop()` with a clear Chinese message telling the learner which earlier script or section to run;
- default paths should use the formal local project directory and local output conventions;
- Biomni trace paths, server paths, downloaded evidence folders, and copied Biomni output tables must never be used as runtime shortcuts;
- downstream-facing output filenames and table schemas should remain stable and close to Biomni corrected outputs so later scripts can consume them.

### 4. Apply decision gates

Use facts from the trace and repo directly. Ask the user only for high-impact choices.

Read `references/decision-gates.md` when a decision may affect scope, analysis behavior, user time, or downstream compatibility.

### 4A. Prepare the beginner environment Word guide

Before writing or extending the reproducible analysis scripts, confirm that the formal local project has a Word `.docx` environment guide for zero-bioinformatics and zero-coding learners.

Do not force R-only conventions onto every project. Use the language and execution style proven by the downloaded trace and local repo. A project may be R-based, Python-based, notebook-based, shell-heavy, or mixed-language.

The guide should be written in Chinese and should explain, at minimum:

- which software to download and why it is needed, such as R/RStudio/Positron/Rtools for R projects, Python/Conda/venv/Jupyter/VS Code for Python projects, Git/GitHub tools when version control is needed, shell tools when command-line execution is required, and any external bioinformatics software used by Biomni;
- the exact software versions, package versions, seeds, and system assumptions extracted from Biomni when available;
- how to create/open the project in the appropriate tool: `.Rproj` for RStudio/Positron R projects, `environment.yml`, `requirements.txt`, `pyproject.toml`, virtual environments, or notebooks for Python projects, and equivalent launcher/config files for other runtimes;
- how local folders such as `data/`, `results/`, `figures/`, `logs/`, notebooks, scripts, and teaching-script folders are used, and why scripts must use the local project path rather than Biomni paths;
- how to install or check required packages without silently changing analysis versions, for example R packages with `renv`/`install.packages()`/Bioconductor when appropriate, Python packages with `conda`, `pip`, `environment.yml`, or `requirements.txt` when appropriate, and external tools with their own version checks;
- how to run scripts or notebooks section by section, why expensive workflows should not be run with one-click Run All / Run All Cells / full pipeline execution, and what files should appear after each major script;
- how to save and restore the working state in a language-appropriate way: `.RData`, `.Rhistory`, and `.Rproj` for R; `.ipynb` notebook state, conda/venv environment files, serialized objects such as `.pkl`/`.joblib` only when appropriate, and command history or shell scripts for Python/CLI projects; and when to prefer reproducible reloads such as `readRDS()` / `fread()` / `read_csv()` / `pickle.load()` over relying only on an interactive session panel;
- common beginner errors, such as wrong working directory, missing package, non-ASCII path issues, missing previous-step outputs, and accidentally reading Biomni trace files as runtime inputs.

If the `.docx` guide is missing, outdated, or inconsistent with the trace/software versions/local project layout, create or revise it before continuing script work. The guide is a teaching prerequisite, but it must not replace script-level `stop()` checks or detailed comments.

### 5. Write the local reproduction script

The script must:

- use local project paths, not Biomni paths;
- require raw/original input files to be local user-provided files;
- require intermediate inputs to be generated by earlier local scripts;
- depend on earlier local outputs with explicit `stop()` messages when missing;
- run the final corrected path by default;
- explain code in Chinese for a zero-bioinformatics and zero-coding reader;
- leave no unexplained executable code lines; add adjacent or inline Chinese comments for every meaningful line;
- preserve Biomni's package versions, software versions, parameters, and code organization as closely as practical;
- make expensive steps default only when the final reproduction requires them, while warning the reader not to Run All;
- keep deprecated/error material as `#` comments only.

### 5A. Beginner Teaching and Comment Layout Contract

Write for a reader who has no bioinformatics background and no coding background.

Every script should be understandable at three levels:

1. **Script overview:** what biological question this script answers, why it matters, what files it reads, what files it writes, and where it sits in the project chain.
2. **Section overview:** before long code blocks, add a visually clear `本节速览` block that states:
   - purpose;
   - inputs;
   - outputs;
   - algorithm or statistical method;
   - key parameters;
   - checks / how to judge results;
   - what must not be over-interpreted.
3. **Line-level help:** keep concise Chinese comments near meaningful code lines, especially for syntax or operations beginners may not know (`<-`, `if`, `for`, `file.exists()`, `stop()`, `merge()`, `fread()`, `fwrite()`, `%in%`, `data.table` grouping, formulas, object slots, and path construction). Do not turn every line into a repeated wall of text when a section overview already explains the block.

For any specific algorithm or statistical method, add an `算法卡片` at the point where it appears. The card should explain:

- what the algorithm does in plain Chinese;
- what problem it solves in this project;
- what inputs it expects;
- what outputs or scores it produces;
- why its parameters are chosen;
- how to interpret the result;
- limitations and common over-interpretations.

Examples of methods that need an algorithm card when used: inferCNV, CopyKAT, normalization, variable feature selection, scaling, PCA, Harmony, UMAP, kNN graph construction, Leiden/Louvain clustering, AddModuleScore, CytoTRACE2, Spearman correlation, linear model residuals, Fisher exact test, BH/FDR correction, Monocle3, pseudotime, root sensitivity, bootstrap, BBKNN, FindMarkers/Wilcoxon, checksum/MD5, and file-level audit logic.

Formatting guidance:

- Prefer short Chinese paragraphs and bullet-like comment blocks over dense uninterrupted comments.
- Use visible dividers such as `本节速览` and `算法卡片` for long or complex blocks.
- Keep historical Biomni mistakes in comments, clearly labelled as `历史错误提醒 / 不要这样做`.
- Do not write executable branches that reproduce old formulas, read deprecated results, or generate wrong-path outputs.
- Explain algorithms and biological concepts before the code that uses them, not only after the output is generated.

## Validation Checklist

Before handing back work:

- Confirm the latest downloaded trace was used as evidence.
- Confirm all user-provided Biomni share links for the same project were downloaded, inventoried, and audited together unless the user explicitly marked some links as unrelated.
- Confirm the cross-trace chronology explains how multiple Biomni agent windows relate to each other before choosing the final corrected route.
- Confirm raw viewer/API responses were saved, including the public replay JSON, results metadata JSON, source URL, API base, and file inventory when available.
- Confirm conclusions and implementation choices are grounded in downloaded local context, execution trace, files, notes, logs, and output tables, not guesses.
- Confirm the script follows Biomni's final corrected route and does not treat the latest version number as final without checking what that version means.
- Confirm Biomni software/package versions were extracted when available and reflected in the script or version checks.
- Confirm software versions, language package versions, random seeds, code flow, and parameters stay close to Biomni unless a documented local-reproduction reason requires changes.
- Confirm original/raw inputs are local user-provided files and not Biomni/public-download runtime dependencies.
- Confirm intermediate inputs are produced by earlier local scripts, not copied from Biomni outputs.
- Confirm a beginner-facing Word `.docx` environment guide exists and covers the correct project language/runtime, software download, installation/configuration, project opening, script/notebook execution, package/version checks, and working-state save/restore.
- Confirm final scripts do not contain Biomni trace paths as runtime defaults.
- Confirm deprecated/error-path terms appear only in comments unless they are part of corrected output column names.
- Confirm the new script's required inputs are produced by earlier local scripts.
- Confirm missing prerequisites stop clearly and tell the user which script to run.
- Confirm changed code blocks have Chinese explanation for every meaningful executable line, while long blocks also have a readable `本节速览` before the code.
- Confirm concrete algorithms/statistical methods have nearby `算法卡片` explanations covering what they do, why used here, inputs/outputs, parameters, interpretation, and limitations.
- Confirm comments are written for absolute beginners and explain both code syntax and biological/research logic in Chinese.
- Run static checks. If the relevant language runtime is available, run parse/syntax checks only unless the user explicitly requested real analysis execution.
