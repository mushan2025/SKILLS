# Biomni Trace Local Repro

Use this skill when the user wants to convert a Biomni share trace into local, reproducible, teaching-oriented bioinformatics scripts.

This skill complements `general-bioinformatics-teaching-script-enhanced`: this skill governs trace download, corrected-path selection, continuity across project scripts, and decision gates; the teaching-script skill governs how to explain code for zero-bioinformatics and zero-coding readers.

## Non-Negotiable Rules

1. **Download first, analyze second.** If the user provides a Biomni share link, download the execution trace, code, manifests, notes, tables, figures, and other available files before judging the analysis.
2. **Use the latest local trace download.** If multiple downloads exist for the same project, prefer the newest one unless the user explicitly asks for an older trace.
3. **Trace paths are evidence only.** Final local reproduction scripts must not use `share_*`, `.tmp_*`, `/mnt/...`, Biomni workspace paths, or downloaded trace folders as default runtime inputs.
4. **Corrected path only.** Executable code must reproduce Biomni's final corrected path. Historical mistakes, deprecated outputs, old formulas, and wrong interpretations must appear only in `#` comments.
5. **Preserve project continuity.** Before writing a new script in an ongoing project, read the earlier local scripts and their outputs so the new script connects cleanly.
6. **Local-uploaded originals only.** Raw/original research input files must come from local user-provided files, not from Biomni runtime paths or automatically downloaded public databases, unless the user explicitly approves a different source.
7. **Local-generated intermediates only.** Any intermediate file required as script input must be generated by earlier local reproduction scripts. Do not use Biomni output tables as default intermediate inputs.
8. **Do not run full workflows by default.** Codex writes code, writes Chinese comments, and performs static/parse checks. Do not execute the complete analysis pipeline unless the user explicitly asks for real execution.
9. **Chinese comments for every code line.** User-facing reproduction scripts should use Chinese comments and Chinese explanatory text by default. Do not leave executable code lines unexplained; each code line needs an adjacent or inline Chinese explanation unless it is purely structural punctuation.
10. **Version and code fidelity.** Software versions, package versions, parameters, and code structure should match Biomni as closely as practical. Change only what is needed for local paths, local prerequisites, teaching clarity, and removal of executable error paths.
11. **Ask for major decisions.** If a choice changes the analysis path, candidate definition, default runtime behavior, output interface, naming convention, script deprecation, deletion/overwrite behavior, whether to run expensive analysis, whether to use non-local source files, or whether to deviate from Biomni versions/code structure, ask the user first.

## Workflow

### 1. Download the Biomni share

- Extract the share token or stable identifier from the user-provided URL.
- Find the real downloadable content behind the share page before analyzing. Read `references/link-discovery.md` for the required discovery procedure.
- Prefer running `scripts/download_biomni_share.py` first when Python and network access are available; it saves viewer/API responses, discovered URLs, reachable files, and a manifest.
- Save the download under a project-local trace folder such as:
  - `biomni_traces/<share_token>_<YYYYMMDD>/`
  - or the nearest existing project convention if one is already present.
- Download all reachable execution trace materials: trace JSON/markdown, code snippets, result metadata, file manifests, tables, figures, notes, and logs.
- Record the source URL, download date, token, and file inventory.
- If network access is blocked, request approval through the active Codex permission flow.

### 2. Audit the trace

Read enough trace material to establish:

- the final corrected analysis path;
- which files are canonical outputs;
- which files are deprecated or historical;
- which formulas, thresholds, candidate definitions, and interpretations were corrected;
- which local files must be generated by earlier scripts.

Never infer the final path from filenames alone. Prefer trace notes, manifests, result metadata, and corrected output tables.

### 3. Inspect existing local scripts

Before creating or extending a script, inspect the same project's existing local reproduction scripts.

Build a mental interface map:

- script order, for example `F0 -> F1 -> F2A -> F2B -> F2C -> F2step4`;
- default `PROJECT_DIR`;
- inputs and outputs of each script;
- object/table names used as handoff points;
- stop conditions and user-facing error style;
- terminology and biological narrative already introduced.

The new script must not create a logical gap. It should neither repeat previous modules as if they did not exist nor assume knowledge that earlier scripts never introduced.

### 4. Apply decision gates

Use facts from the trace and repo directly. Ask the user only for high-impact choices.

Read `references/decision-gates.md` when a decision may affect scope, analysis behavior, user time, or downstream compatibility.

### 5. Write the local reproduction script

The script must:

- use local project paths, not Biomni paths;
- require raw/original input files to be local user-provided files;
- require intermediate inputs to be generated by earlier local scripts;
- depend on earlier local outputs with explicit `stop()` messages when missing;
- run the final corrected path by default;
- explain code in Chinese for a zero-bioinformatics and zero-coding reader;
- leave no unexplained executable code lines; add adjacent or inline Chinese comments for every meaningful line;
- preserve Biomni's package versions, software versions, parameters, and code organization as closely as practical;
- make expensive steps default only when the final reproduction requires them, while warning the reader not to Run All;
- keep deprecated/error material as `#` comments only.

For detailed contract rules, read `references/repro-contract.md`.

For instructions on finding the downloadable contents behind a Biomni share URL, read `references/link-discovery.md`.

For continuity rules, read `references/project-continuity.md`.

For the STAD F2 example, read `references/stad-f2-case.md`.

## Validation Checklist

Before handing back work:

- Confirm the latest downloaded trace was used as evidence.
- Confirm Biomni software/package versions were extracted when available and reflected in the script or version checks.
- Confirm code flow and parameters stay close to Biomni unless a documented local-reproduction reason requires changes.
- Confirm original/raw inputs are local user-provided files and not Biomni/public-download runtime dependencies.
- Confirm intermediate inputs are produced by earlier local scripts, not copied from Biomni outputs.
- Confirm final scripts do not contain Biomni trace paths as runtime defaults.
- Confirm deprecated/error-path terms appear only in comments unless they are part of corrected output column names.
- Confirm the new script's required inputs are produced by earlier local scripts.
- Confirm missing prerequisites stop clearly and tell the user which script to run.
- Confirm changed code blocks have Chinese explanation for every meaningful executable line.
- Run static checks. If R is available, run parse/syntax checks only unless the user explicitly requested real analysis execution.
